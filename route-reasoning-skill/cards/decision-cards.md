# Decision Cards

每张 Decision Card 用来支撑一次路线判断。生成 RoutePlan 时至少选择 3 张卡。

## 1. 需求不清先规划

- 名称：需求不清先规划
- 来源领域：项目管理
- 触发条件：用户目标模糊、范围不明、验收标准缺失
- 判断规则：先产出 RoutePlan，不直接生成实现指令
- 输出影响：推荐 Agent 为 Claude Code 或 Human
- 适用场景：探索型需求、产品方向不确定
- 不适用场景：明确 bug 和明确文件级修复

## 2. 前提不足不下结论

- 名称：前提不足不下结论
- 来源领域：逻辑学
- 触发条件：关键事实缺失或上下文只是一部分
- 判断规则：列出 assumptions，避免断言完整项目状态
- 输出影响：降低 confidence，增加人工确认节点
- 适用场景：用户只提供片段、Snapshot 不完整
- 不适用场景：用户明确提供完整验收输入

## 3. 约束优先

- 名称：约束优先
- 来源领域：风险控制
- 触发条件：用户明确说不要改某区域、不要新增依赖、不要执行命令
- 判断规则：约束优先于速度和功能扩展
- 输出影响：写入 forbiddenChanges
- 适用场景：生产项目、发布前、小修复
- 不适用场景：探索性 demo 且用户允许大改

## 4. 小步可验证优先

- 名称：小步可验证优先
- 来源领域：运筹学
- 触发条件：任务可拆分，且每步能单独验证
- 判断规则：先做最小闭环，再扩展
- 输出影响：Route 拆成 P0/P1/P2
- 适用场景：bugfix、UI cleanup、小功能
- 不适用场景：必须一次性完成的迁移

## 5. 高风险先分析

- 名称：高风险先分析
- 来源领域：风险控制
- 触发条件：涉及架构、数据、认证、Provider、发布流程
- 判断规则：先让 Claude Code 分析，不直接改代码
- 输出影响：推荐 Agent 为 Claude Code 或 Reviewer
- 适用场景：重构、迁移、安全相关任务
- 不适用场景：低风险文案修改

## 6. 成本受限选低成本路径

- 名称：成本受限选低成本路径
- 来源领域：运筹学
- 触发条件：用户关心 token、时间、费用或只要快速判断
- 判断规则：选择小模型、规则路径或更短任务包
- 输出影响：推荐 DeepSeek 或规则解析
- 适用场景：小范围明确任务
- 不适用场景：高风险架构决策

## 7. 稳定优先

- 名称：稳定优先
- 来源领域：风险控制
- 触发条件：用户担心改坏现有功能
- 判断规则：优先保持现有行为，不做无关优化
- 输出影响：验收标准加入“不破坏现有功能”
- 适用场景：bugfix、release、老项目维护
- 不适用场景：允许重写的原型项目

## 8. UI 变更隔离

- 名称：UI 变更隔离
- 来源领域：项目管理
- 触发条件：任务涉及 UI 或用户明确不希望 UI 被改乱
- 判断规则：UI 修改必须单独列范围，避免顺手改布局
- 输出影响：forbiddenChanges 加入无关 UI 禁止项
- 适用场景：UI cleanup、bugfix
- 不适用场景：整体视觉改版

## 9. 重构先读项目

- 名称：重构先读项目
- 来源领域：项目管理
- 触发条件：用户提出架构、拆分、整理、重构
- 判断规则：先分析结构和依赖，再生成修改方案
- 输出影响：推荐 Claude Code，默认先分析不改代码
- 适用场景：模块拆分、技术债治理
- 不适用场景：单行 bugfix

## 10. 明确 bug 交给 Codex

- 名称：明确 bug 交给 Codex
- 来源领域：项目管理
- 触发条件：bug 现象、目标和约束都清楚
- 判断规则：直接生成最小修复指令
- 输出影响：推荐 Codex
- 适用场景：报错、测试失败、交互异常
- 不适用场景：根因完全未知且影响大

## 11. 多方案先会审

- 名称：多方案先会审
- 来源领域：逻辑学
- 触发条件：存在多个可行方案且取舍影响较大
- 判断规则：先比较方案，不立即选实现
- 输出影响：推荐 Reviewer 或 multi-review
- 适用场景：架构选型、发布策略、迁移路线
- 不适用场景：明确唯一修复路径

## 12. 发布前先检查

- 名称：发布前先检查
- 来源领域：风险控制
- 触发条件：用户准备发版、开源、上线或交付
- 判断规则：先做安全、文档、构建、隐私检查
- 输出影响：生成 release checklist
- 适用场景：GitHub release、生产上线
- 不适用场景：早期原型探索

## 13. 可回滚优先

- 名称：可回滚优先
- 来源领域：风险控制
- 触发条件：改动可能影响用户数据或核心流程
- 判断规则：优先选择容易回退的方案
- 输出影响：验收标准加入回滚或复原方式
- 适用场景：配置、数据、发布
- 不适用场景：纯文档更新

## 14. 依赖最少优先

- 名称：依赖最少优先
- 来源领域：运筹学
- 触发条件：方案需要新增库、服务或外部工具
- 判断规则：能不用依赖就不用；必须新增时写理由
- 输出影响：constraints 加入“不新增依赖，除非确认”
- 适用场景：前端、脚本、后端小修
- 不适用场景：用户明确要求引入特定依赖

## 15. 数据变更先备份

- 名称：数据变更先备份
- 来源领域：风险控制
- 触发条件：涉及数据库、迁移、用户记录
- 判断规则：先备份和验证迁移方案，再执行修改
- 输出影响：推荐 Human 确认或 Claude Code 分析
- 适用场景：schema migration、数据清理
- 不适用场景：无状态 UI 修改

## 16. 安全边界先声明

- 名称：安全边界先声明
- 来源领域：风险控制
- 触发条件：任务涉及命令执行、密钥、权限、认证
- 判断规则：先声明不读取密钥、不自动执行、不越权
- 输出影响：forbiddenChanges 加入安全边界
- 适用场景：Provider、auth、deployment
- 不适用场景：普通文案调整

## 17. 上下文不足先要摘要

- 名称：上下文不足先要摘要
- 来源领域：逻辑学
- 触发条件：任务依赖项目结构但用户未提供上下文
- 判断规则：请求文件树或关键文件摘要，不臆测
- 输出影响：增加 Handoff 的 missingContext
- 适用场景：跨模块问题、架构判断
- 不适用场景：用户只需要通用建议

## 18. 先验收后扩展

- 名称：先验收后扩展
- 来源领域：项目管理
- 触发条件：用户提出多个追加想法
- 判断规则：先完成当前目标验收，再安排扩展
- 输出影响：P0 只保留必要任务
- 适用场景：feature、UI cleanup
- 不适用场景：用户要求一次性设计完整路线

## 19. 高影响决策人工确认

- 名称：高影响决策人工确认
- 来源领域：风险控制
- 触发条件：改动不可逆、影响数据、影响发布或权限
- 判断规则：执行前必须人工确认
- 输出影响：推荐 Agent 为 Human 或加入确认门
- 适用场景：删除、迁移、发布、权限
- 不适用场景：局部测试修复

## 20. Handoff 保留决策理由

- 名称：Handoff 保留决策理由
- 来源领域：项目管理
- 触发条件：任务跨多轮完成
- 判断规则：交接摘要必须写明为什么选当前路线
- 输出影响：Handoff 包含 decisions 和 rationale
- 适用场景：长期项目、多 Agent 协作
- 不适用场景：一次性小任务

## 21. 执行和配置分层

- 名称：执行和配置分层
- 来源领域：逻辑学
- 触发条件：任务同时涉及执行 Agent 和 Provider / 环境配置
- 判断规则：执行 Agent 负责代码任务，配置工具只负责环境或 Provider 配置
- 输出影响：任务包分成执行指令和配置说明
- 适用场景：本地 AI 工具链、Provider 切换
- 不适用场景：单纯代码实现

## 22. 目标函数先定义

- 名称：目标函数先定义
- 来源领域：运筹学
- 触发条件：用户同时追求速度、质量、成本、稳定性、长期扩展或多个产品目标
- 判断规则：先明确本轮主要优化目标，再比较路线；目标函数不清时先问选择题
- 输出影响：RoutePlan 的 Goal 和 Options 必须使用同一目标函数比较
- 适用场景：路线选择、资源受限、产品方向取舍
- 不适用场景：目标和验收已经明确的小修复

## 23. 硬约束不可违反

- 名称：硬约束不可违反
- 来源领域：运筹学 / 风险控制
- 触发条件：用户明确限制范围、依赖、预算、时间、权限、数据、UI 或执行方式
- 判断规则：任何违反硬约束的路线直接淘汰，不进入最终推荐
- 输出影响：Options 标记 eliminated 或在 Recommended Route 说明淘汰原因
- 适用场景：发布前、生产环境、小范围修改、预算敏感任务
- 不适用场景：用户明确要求开放探索且没有硬限制

## 24. 事实和假设分离

- 名称：事实和假设分离
- 来源领域：逻辑学
- 触发条件：用户只给了片段、猜测、二手描述、截图、摘要或不完整上下文
- 判断规则：所有结论必须区分 confirmed facts 和 assumptions
- 输出影响：Situation 或 Risks 必须列出“已知事实 / 假设 / 待确认”
- 适用场景：项目诊断、任务交接、Agent 输出审查
- 不适用场景：用户提供完整、可验证、无歧义输入

## 25. 概念不清先定义

- 名称：概念不清先定义
- 来源领域：逻辑学
- 触发条件：用户使用“优化”“完善”“做得更好”“重构”“有效果”等宽泛词
- 判断规则：先把宽泛词转成可验收定义，再生成 TaskSpec
- 输出影响：Choice Questions 或 TaskSpec 必须包含成功定义
- 适用场景：产品规划、写作、学习路线、体验优化
- 不适用场景：错误信息、目标、验收都已经明确

## 26. 推荐必须有理由链

- 名称：推荐必须有理由链
- 来源领域：逻辑学
- 触发条件：需要推荐一条最终路线
- 判断规则：推荐必须串联目标、约束、风险、评分和第一步
- 输出影响：Recommended Route 必须包含 Reason Chain
- 适用场景：所有 RoutePlan
- 不适用场景：只生成选择题且尚未进入路线推荐

## 27. 防止工具光环

- 名称：防止工具光环
- 来源领域：博弈论 / 决策心理学
- 触发条件：用户指定某 Agent、模型或工具，但任务类型未必匹配
- 判断规则：先判断任务激励和能力边界，再分配执行者；不要因为工具强就默认推荐
- 输出影响：Selected Agent 必须说明适配原因和不适配的替代 Agent
- 适用场景：Codex / Claude Code / DeepSeek / Human / Reviewer 分工
- 不适用场景：用户明确要求单一工具且风险很低

## 28. 激励一致才委派

- 名称：激励一致才委派
- 来源领域：博弈论
- 触发条件：任务要交给人、团队成员、外部 Agent 或自动化工具
- 判断规则：执行者目标、可见信息、权限和验收标准必须一致；不一致时先设置检查点
- 输出影响：TaskSpec 增加 Report Back、Acceptance Criteria 和 Review 节点
- 适用场景：多 Agent 协作、团队任务分派、外包式指令
- 不适用场景：用户自己立即执行的单步任务

## 29. 最小后悔优先

- 名称：最小后悔优先
- 来源领域：博弈论 / 决策理论
- 触发条件：信息不足但必须先行动，且不同路线失败代价不同
- 判断规则：优先选择未来后悔成本最低、最容易回退和补救的第一步
- 输出影响：First Action 选择低风险、可回滚、可复核动作
- 适用场景：需求不清、高风险决策、发布前判断
- 不适用场景：已经有强证据支持唯一方案

## 30. 瓶颈优先

- 名称：瓶颈优先
- 来源领域：运筹学
- 触发条件：某个缺口阻止后续开发、测试、判断、发布或学习
- 判断规则：先处理瓶颈任务，再做锦上添花的优化
- 输出影响：Priority 标记 P0，First Action 指向瓶颈验证
- 适用场景：测试失败、关键上下文缺失、构建阻塞、决策输入缺失
- 不适用场景：多个任务互不依赖且价值差异明显

## 31. 便宜模型先证伪

- 名称：便宜模型先证伪
- 来源领域：运筹学 / 成本控制
- 触发条件：任务可以拆成可验证步骤，且高价模型不是完成第一步的必要条件
- 判断规则：默认先让 Sonnet 完成第一轮；只有出现失败证据、重大风险或无法解决的歧义时才升级
- 输出影响：ModelCabinetPlan 必须写出 Sonnet 的第一任务和升级闸门
- 适用场景：实现、修改、整理、测试、常规分析
- 不适用场景：不可逆高风险决策或用户明确要求最高级别裁决

## 32. 贵模型只解瓶颈

- 名称：贵模型只解瓶颈
- 来源领域：运筹学
- 触发条件：Opus 或 Fable 5 成本显著高于默认执行模型
- 判断规则：高价模型只接收阻塞后续工作的最小上下文与明确问题，不接收整段重复执行任务
- 输出影响：每个高价模型调用必须包含 Bottleneck、Expected Decision 和 Stop Condition
- 适用场景：困难调试、架构冲突、安全复核、战略裁决
- 不适用场景：无法拆分且必须全局阅读的任务

## 33. 裁决后立即降级

- 名称：裁决后立即降级
- 来源领域：博弈论 / 委派机制
- 触发条件：Fable 5 或 Opus 已经给出清晰路线、约束和验收标准
- 判断规则：Fable 5 作为首辅时只负责委派与裁决；将后续执行交还 Sonnet，高价模型只保留复核权，不继续代替执行者
- 输出影响：ModelCabinetPlan 必须包含 Delegation Calls、Downshift 和 Review Checkpoint
- 适用场景：模型内阁、复杂任务拆分、成本敏感工作流
- 不适用场景：执行本身仍需要高价模型独有能力且无法验证拆分

## 34. 独立表态防从众

- 名称：独立表态防从众
- 来源领域：博弈论 / 群体决策
- 触发条件：多个成员、Agent 或模型参与同一决策
- 判断规则：成员必须先在不知道其他答案的情况下提交路线、理由、证据和风险，再进入交叉质询
- 输出影响：DecisionReceipt 必须记录独立意见和 Strongest Dissent
- 适用场景：模型会审、高价值决策、方案评审
- 不适用场景：单执行者的明确小任务

## 35. 否决必须可解除

- 名称：否决必须可解除
- 来源领域：风险控制 / 机制设计
- 触发条件：路线可能违反硬约束、安全边界或造成不可接受的不可逆损失
- 判断规则：Veto 必须引用明确依据，并给出解除否决所需的证据、修复或人工确认
- 输出影响：Veto Record 必须包含 basis、impact 和 unblock condition
- 适用场景：安全、权限、迁移、删除、发布、重大投入
- 不适用场景：普通偏好分歧或低风险可回滚任务

## 36. 先写翻案条件

- 名称：先写翻案条件
- 来源领域：逻辑学 / 贝叶斯更新
- 触发条件：推荐路线依赖假设、预测或不完整信息
- 判断规则：裁决时预先写出哪些可观察证据会推翻当前推荐，并指定触发后的动作
- 输出影响：DecisionReceipt 必须包含至少 1 条 Flip Condition
- 适用场景：产品路线、架构、发布、研究、资源分配
- 不适用场景：完全确定且无替代路线的机械任务

## 37. 结果校准而非结果论

- 名称：结果校准而非结果论
- 来源领域：决策理论
- 触发条件：决策执行后已有真实结果
- 判断规则：分别评价决策过程质量与执行质量；好结果不自动证明决策正确，坏结果也不自动证明决策错误
- 输出影响：OutcomeReview 必须区分 Decision Quality 和 Execution Quality
- 适用场景：项目复盘、发布复盘、模型建议命中率
- 不适用场景：尚未执行或没有观察结果
