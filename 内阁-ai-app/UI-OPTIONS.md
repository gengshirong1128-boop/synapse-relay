# 内阁 UI 整体形态方案

本文件只讨论阶段 C/D 之前的产品形态选择，不涉及代码实现。三套方案都遵守两个已定决策：

- 点 agent 头像打开配置抽屉/侧板，用于切模型、开关 skill、开关插件、调整角色；配置不进入聊天气泡。
- yes/no 审批留在对话流中，以醒目的审批卡片呈现，提供「允许一次 / 本会话允许 / 拒绝」。

整体视觉基调建议延续阶段 B 的深色专业工具方向：克制、低装饰、强层级，古风命名只作为角色标签，不把界面做成古风皮肤。

## 方案甲：群聊式三栏

一句话定位：最像 Discord/Slack 的协作工作台，适合习惯用会话组织工作的用户。

适合场景：用户经常在私聊和群聊之间切换，需要持续追踪多个 agent 的发言、状态和权限边界。

### 布局示意

```text
+----------------------+--------------------------------------+---------------------------+
| 会话列表              | 当前会话                              | 成员与分工                 |
|----------------------|--------------------------------------|---------------------------|
| 检测本地工具          | 顶部：会话标题 / 当前目标 / 状态      | 成员列表                   |
| + 新建群聊            |--------------------------------------| - Codex     thinking      |
|                      | 消息流                                | - Claude    waiting       |
| 私聊                  |  agent 头像 + 名称 + 角色 + 内容      | - Gemini    idle          |
| - Codex               |  审批卡片                             |                           |
| - Claude              |  边界提醒 / 监工事件                  | 首辅分工                   |
|                      |                                      | 工作区 / read_paths        |
| 群聊                  | 底部输入区                            | write_paths / 验收标准     |
| - 前端重构内阁        | @agent / 附加约束 / 发送              |                           |
| - 后端接入审查        |                                      | 边界报告 / 验收 / 监工      |
+----------------------+--------------------------------------+---------------------------+
```

### 核心难点如何解决

多 agent 同时发言：
消息流仍按时间排列，但每条消息左侧固定显示头像、agent 名称、角色标签和状态小点。连续同一 agent 发言做轻量合并，减少刷屏。顶部增加「等待链」提示，例如 `Claude 等 Codex 输出文件清单`，让用户知道谁在等谁。

私聊与群聊切换：
左栏分成「私聊」和「群聊」两组。私聊是单 agent 会话，群聊是任务/议题会话。群聊列表项显示成员头像叠层、未处理审批数、当前 wave 状态。

拉人/踢人：
右栏成员区顶部放「添加成员」按钮；成员行右侧有更多菜单，可踢出、静音、调整角色。拉人后在消息流产生一条系统事件，避免成员变化无痕发生。

首辅分工：
右栏下半部分是首辅分工面板。点击成员后显示该成员的工作区、可读路径、可写路径、验收标准和依赖关系。复杂编辑使用右栏内联表单，不打断聊天。

头像配置抽屉：
点击任意头像，从右侧覆盖打开 agent 配置抽屉；抽屉层级高于成员面板，但不离开当前会话。抽屉关闭后回到原消息位置。

对话流审批卡片：
审批卡片插在触发请求的上下文之后，卡片顶部写明发起 agent、请求动作、影响范围和风险等级。按钮固定为「允许一次 / 本会话允许 / 拒绝」，拒绝后可填写一句原因并回写到对话。

边界报告 / 验收 / 监工：
右栏底部做成「审查」小节，展示最新边界报告、验收结果、监工事件。重要异常同时进入消息流，以免用户只盯中栏时错过。

### 优点

- 心智成本最低，用户容易理解「左边选会话、中间聊天、右边管成员」。
- 私聊/群聊切换自然，适合持续使用。
- 现有边界、验收、监工数据能平滑放进右栏，不需要重塑数据模型。

### 缺点

- 多 agent 高并发输出时，中栏仍可能变长，需要合并、折叠和过滤策略。
- 右栏承载成员、分工、审查三类信息，信息密度容易偏高。
- 在窄屏上三栏会退化明显，需要设计好抽屉层级。

实现复杂度：中。

移动端/窄屏退化：
默认只显示当前会话消息流。左栏变成顶部会话切换按钮，右栏变成「成员/分工」底部抽屉。审批卡片仍留在消息流，不变成 Modal。

## 方案乙：协作看板 / 议事厅式

一句话定位：把多 agent 协作呈现为一个会议议题下的并行工作板，适合偏项目管理和审查流的用户。

适合场景：用户更关心任务推进、分工状态和结果汇总，而不是像聊天工具一样逐条读消息。

### 布局示意

```text
+--------------------------------------------------------------------------+
| 议题标题 / 目标 / 当前 wave / 全局审批数 / 成员入口                       |
+--------------------------------------------------------------------------+
| Agent 列 A              | Agent 列 B              | Agent 列 C            |
|-------------------------|-------------------------|-----------------------|
| Codex                   | Claude                  | Gemini                |
| 当前状态：执行中        | 当前状态：等待 Codex    | 当前状态：空闲        |
|                         |                         |                       |
| 发言卡片                | 发言卡片                | 发言卡片              |
| 文件变更请求审批卡      | 等待原因卡              | 建议卡                |
| 输出摘要                | 审查意见                |                       |
+--------------------------------------------------------------------------+
| 首辅决策区：分工 / 边界 / 验收标准 / 拉人踢人 / 边界报告 / 监工事件       |
+--------------------------------------------------------------------------+
```

### 核心难点如何解决

多 agent 同时发言：
每个 agent 有独立列，天然区分身份。列头显示状态：发言中、等待谁、需要审批、已完成。新发言进入对应列，不会把其他 agent 的上下文冲走。跨 agent 依赖用细线或「等待原因卡」表达。

私聊与群聊切换：
私聊是单列视图，群聊是多列议事厅。左侧不一定常驻会话列表，可以放在顶部的会话切换器中，减少横向占用。

拉人/踢人：
顶部成员入口或底部首辅决策区提供「拉入成员」。拉入后新增一列；踢出后列折叠为历史摘要，不直接消失，保留协作记录。

首辅分工：
底部固定「首辅决策区」最适合承载分工。用户可以按 agent 切换表单，也可以在表格中横向比较每个 agent 的工作区、边界和验收标准。

头像配置抽屉：
点击列头头像打开配置侧板。侧板从右侧覆盖，不改变列布局；侧板中只管单 agent 的模型、skill、插件和角色。

对话流审批卡片：
审批卡片出现在对应 agent 列中，同时在顶部显示全局审批计数。这样审批保留上下文，又不会被其他 agent 发言淹没。

边界报告 / 验收 / 监工：
底部首辅决策区可分成三个 tab：「分工」「审查」「监工」。边界报告适合表格化展示，验收适合 checklist，监工适合事件流。

### 优点

- 多 agent 并发最清楚，身份和等待关系比聊天流更稳定。
- 很适合展示分工、边界和验收，符合「首辅调度」的产品核心。
- 审批卡片不容易被刷走。

### 缺点

- 对纯聊天用户不如三栏自然，学习成本略高。
- agent 数量超过 4 个后横向空间压力大，需要列折叠和摘要策略。
- 私聊到群聊的心智切换比方案甲更明显。

实现复杂度：高。

移动端/窄屏退化：
多列改为横向 tab：顶部选择 agent，主体显示当前 agent 列。底部首辅决策区变成可拖拽底部面板。全局审批卡片可置顶显示，但实际决策仍回到对应 agent 的上下文卡片。

## 方案丙：任务中枢 + 会话流

一句话定位：以「当前目标」为中心，把聊天流、成员调度和审查结果合成一个可折叠的任务中枢，适合既要聊天感又要项目推进感的用户。

适合场景：用户希望主界面干净，不想长期看三栏，也不想把 agent 发言拆成过多列；更像 Linear/Raycast 式的专注工作台。

### 布局示意

```text
+------------------------------+--------------------------------------------+
| 目标中枢                     | 会话流                                     |
|------------------------------|--------------------------------------------|
| 当前目标                     | 顶部：会话名 / 私聊或群聊 / 成员叠层       |
| 进度 / wave / 风险           |--------------------------------------------|
|                              | 关键消息                                   |
| 等待关系                     | agent 发言摘要                             |
| Codex -> Claude              | 展开的详细发言                             |
| Claude -> 都察院             | 审批卡片                                   |
|                              | 边界异常提醒                               |
| 成员与分工                   |                                            |
| agent 列表                   | 底部输入区                                 |
| 工作区 / 边界 / 验收摘要     |                                            |
|                              |                                            |
| 审查结果                     |                                            |
| 边界 / 验收 / 监工           |                                            |
+------------------------------+--------------------------------------------+
```

### 核心难点如何解决

多 agent 同时发言：
会话流不显示所有原始输出，而是默认显示「摘要消息 + 可展开详情」。当多个 agent 同时输出时，系统先形成一个「并行进展组」，组内按 agent 分段，用户可以展开某个 agent 的完整输出。

私聊与群聊切换：
顶部会话名旁边有 `Private / Group` 切换标识，左侧目标中枢只展示当前目标相关会话。会话列表不常驻，改为快捷切换器或命令菜单，减少界面重量。

拉人/踢人：
在左侧成员与分工模块操作。拉人不是创建新列，而是加入当前目标的成员列表；踢人时保留其历史贡献摘要。适合较克制的单主界面。

首辅分工：
左侧目标中枢承担首辅分工。成员列表每行显示工作区、边界状态、验收进度摘要；点击成员在左侧展开编辑，或打开右侧配置抽屉做更完整设置。

头像配置抽屉：
在会话流和左侧成员列表中点击头像都打开配置抽屉。抽屉用于 agent 个体配置，左侧中枢用于任务级分工，二者边界清楚。

对话流审批卡片：
审批卡片仍在右侧会话流中，但会同步在左侧「风险/待决策」区域计数。用户可以从左侧点进对应审批卡片位置。

边界报告 / 验收 / 监工：
左侧底部做「审查结果」模块，显示三个摘要数字和最新事件；点击后在会话流中定位到详细报告卡，或展开左侧详情。

### 优点

- 比三栏更安静，主界面更像专业工作台。
- 比议事厅更保留聊天感，不会把所有对话拆散。
- 等待关系、风险、验收进度常驻在左侧，适合长任务。

### 缺点

- 摘要/展开机制需要设计得很准，否则用户会担心漏看细节。
- 会话列表弱化后，多项目切换效率不如方案甲。
- 需要更强的信息折叠策略，对文案和状态设计要求高。

实现复杂度：中到高。

移动端/窄屏退化：
默认显示会话流，顶部增加「目标中枢」按钮。点击后打开全屏面板，包含等待关系、成员分工、边界报告、验收和监工。审批卡片仍留在会话流原位。

## 横向对比

| 维度 | 方案甲：群聊式三栏 | 方案乙：议事厅看板 | 方案丙：任务中枢 + 会话流 |
| --- | --- | --- | --- |
| 上手难度 | 低 | 中 | 中 |
| 多 agent 并发清晰度 | 中 | 高 | 中到高 |
| 私聊/群聊组织 | 高 | 中 | 中 |
| 首辅分工表达 | 中 | 高 | 高 |
| 审批上下文保留 | 高 | 高 | 高 |
| 边界/验收/监工呈现 | 中 | 高 | 高 |
| 窄屏适配 | 中 | 中到低 | 高 |
| 实现复杂度 | 中 | 高 | 中到高 |

## 我的推荐

如果下一步目标是尽快把阶段 C 做成可用体验，推荐先选方案甲。它最稳，和用户对「私聊/群聊」的认知最贴近，也最容易把现有后端数据接进去。

如果更想突出「多 agent 协作调度」这个差异化能力，可以选方案丙作为产品方向。它比三栏更有自己的性格，但仍保留聊天主线，不像方案乙那样一下子跳到高复杂度。
