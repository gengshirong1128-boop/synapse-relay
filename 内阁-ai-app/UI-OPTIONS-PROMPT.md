## 产品背景
「内阁」是**多 AI agent 协作工作台**（Web）。新方向：
- 检测用户电脑**本地已装的 agent 工具**(Codex / Claude Code / Gemini CLI / OpenCode 等)当「成员/大臣」，**不需要用户填 API key**。
- 用户能**私聊**单个 agent，也能开**群聊**让多个 agent 协作；群聊能**拉人/踢人**。
- **主持人(首辅)**给各 agent 分配**工作区 + 边界(可读/可写路径) + 验收标准**；有**审查**角色(都察院)。
- 古风命名只是角色皮肤，**保持现代专业协作工具观感**(参考 Linear/Discord/Raycast)，国外用户也要看得懂。

## 已确定的两个设计决策（方案必须遵守，不要推翻）
1. **agent 个体配置（切模型 / 开关 skill / 开关插件 / 角色）→ 点 agent 头像弹出"配置抽屉/侧板"**。聊天区保持干净，配置不混进对话气泡。像微信点头像看资料卡。
2. **yes/no 审批（同意 agent 执行命令/改文件）→ 做成对话流里的醒目审批卡片**，带「允许一次 / 本会话允许 / 拒绝」按钮。决策留在对话上下文里，不弹独立 Modal。

## 内阁真实架构（方案必须如实反映，否则是脱离后端的空壳）
后端已有的机制（不可臆造成"像人一样聊天"的样子）：
- **两类成员，本质不同，UI 必须区分**：
  - **执行型 agent 工具**(本地 Codex/Claude Code/Gemini CLI)：通过 subprocess 拉起本地 CLI，**能读写文件、跑命令、真干活**。受限于用户本机装了哪些。
  - **议政型模型**(通过 API/本地工具的模型)：**只能出意见/审查/写方案，不能动文件**。用户可能没几个 agent 工具，但能接很多模型——所以模型当"文官出主意"，agent 工具当"干吏执行"。首辅分工时按此分派。
- **AI 之间不直接对话，靠共享文档协作**：所有成员读写同一个 `plan.json` + `.synapse/collaboration/` 事件日志。首辅写计划→干吏读计划执行→监工读状态写建议。UI 要让用户理解"它们通过共享工作区/计划协作"，而非"在群里互相聊天"。
- **agent 是无状态的，没有跨调用记忆**：每次执行都是全新进程，首辅/干吏**每次都要重读共享文档**才知道上下文。UI 不能给用户"agent 记得之前对话"的错觉；"对话历史"本质是共享文档 + 事件流的可视化。
- **一个协作空间 = 一个持续的 Plan**：群聊会话里可发起多次执行(每次生成 wave/tasks)，不必每次开新对话。

## 需要你设计的：整体形态出 2-3 个方案
每个方案都要解决这些**核心难点**(这是群聊式天然的痛点，重点说清你怎么解)：
- 多 agent 同时发言时，如何让用户**看清谁说了什么、谁在等谁**(身份区分、不刷屏)
- **两类成员(执行型工具 / 议政型模型)在 UI 上怎么区分、怎么各司其职**
- **"群聊对话" 和 "实际执行(读共享文档跑任务)" 在 UI 上怎么衔接**——让用户分得清"在讨论" vs "在干活"
- **私聊 ↔ 群聊**如何切换和组织
- **拉人/踢人**的入口和交互
- **首辅分工**(给成员派工作区/边界/验收)放在哪、怎么操作；区分派给"工具"还是"模型"
- 上面两个已定决策(头像配置抽屉、对话流审批卡片)如何融入
- 现有后端已有的**边界报告 / 验收 / 监工**数据如何呈现(别丢)
- 如何不给用户"agent 有持久记忆"的错觉(执行时重读文档这件事怎么自然呈现)

### 建议至少覆盖这两种方向，再加你自己想的第三种：
- **方向甲：群聊式三栏**（左：私聊/群聊会话列表；中：消息流+审批卡片；右：成员列表+首辅分工面板）
- **方向乙：协作看板/议事厅式**（上：议题/目标；中：各 agent 分列发言或卡片；下：首辅决策区）
- **方向丙：你的 product 直觉**
